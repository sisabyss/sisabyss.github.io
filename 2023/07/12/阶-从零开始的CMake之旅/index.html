「阶」从零开始的CMake之旅

Created2023-07-12|Updated2023-09-10
|Post View:

什么是编译器

编译器, 是一个根据源代码生成机器码的程序。
>g++ main.cpp -o a.out

该命令会调用编译器程序g++，让他读取main.cpp中的字符串（称为源代码），并根据c++标准生成相应的机器指令码，输出至a.out这个文件中，（称为可执行文件）。
>./a.out

之后执行该命令，操作系统会读取刚刚生成的可执行文件，从而执行其中编译成的机器码，调用系统提供的printf函数，并在终端显示出Hello, world。
顺便一提，我们可以看看a.out里面，究竟有什么呢？

可以看到，在main函数中先压栈，然后调用了printf函数。
那么printf在哪呢？其实printf实在/lib/libc.so.6中实现的
为什么他们能够互相调用呢？

其实是在编译过程中，自动链接到libc中的函数。
即为动态链接库。

多文件编译与链接

单文件编译虽然方便，但也有以下缺点：
  1. 所有代码堆在一起，不易于理解和模块化。
  2. 工程量变大时，编译时间将非常长，改动一个地方就得全部重新编译。
    因此，我们提出了多文件编译的概念，文件之间通过符号声明相互引用。
  • >g++ -c hello.cpp -o hello.o
  • >g++ -c main.cpp -o main.o
    其中，我们通过-c选项指定生成临时的对象文件 main.o，再通过一系列对象文件进
    行链接, 最终得到a.out
  • >g++ hello.o main.o -o a.out
    这样，加入我们需要改动hello.cpp,（比如改成Goodbye, world），我们只需要重新编译
    hello.cpp, 最终链接就行了。

为什么需要构建系统(Makefile)

当文件越来越多的时候，一个个调用g++编译链接会非常麻烦。
于是，发明了make这个程序，你只需要写出不同文件的依赖关系，和生成各文件的规则。

>make a.out
敲下这个命令，就可构建出a.out这个可执行文件

假如我们不修改源文件，直接make

你咋没改一个文件呢？不需要编译了。
而假如我们只改变hello.cpp, 再make。

自动地只编译hello, 节省时间。

和直接使用一个脚本写出完整的构建过程相比，make指明依赖关系的好处：

  1. 当更新了一个文件时只会重新编译这个文件, 而不需要重新编译其他文件。
  2. 能够自动并行地发起编译，加快编译速度(make -j)。
  3. 用通配符批量生成构建规则，避免针对每个.cpp和.o重复写g++命令(%.o: %.cpp)。

但缺点也不少：

  1. make在Unix类系统上是通用的，但在Windows则不然。
  2. 需要准确地指明每个项目的依赖关系，加入添加头文件还要修改makefile, 头疼。
  3. make的语法过于简单，不能做出判断。
  4. 不同编译器有不同的flag规则，为g++准备的参数可能对MSVC不适用。

构建系统的构建系统(CMake)

为了解决make的以上问题，跨平台的CMake应运而生！
  1. make在Unix类系统上是通用的，但在Windows则不然。
    只需要写一份CMakeLists.txt, 他就能在调用时生成当前系统所支持的构建系统。
  2. 需要准确地指明每个项目的依赖关系，加入添加头文件还要修改makefile, 头疼。
    CMake可以自动检测源文件和头文件的依赖关系，导出至Makefile里。
  3. make的语法过于简单，不能做出判断。
    CMake具有相对高级的语法，内置的函数能够处理configure，install等常见需求。
  4. 不同编译器有不同的flag规则，为g++准备的参数可能对MSVC不适用。
    CMake可以自动检测当前的编译器，需要添加哪些flag。比如OpenMp，只需要在
    CMakeLists.txt中指明target_link_libraries(a.out OpenMP::OpenMP_CXX)即可

CMake的命令行调用

  1. 读取当前目录的CMakeLists.txt, 并在build文件夹下生成build/Makefile:
    cmake -B build
  2. 让make读取build/Makefile, 并开始构建a.out:
    make -C build
    还可以：
    cmake --build build
  3. 执行生成的a.out:
    build/a.out

Sample

  1. CMakeLists.txt:
  2. cmake -B build
  1. cmake --build build

为什么需要库(library)

  • 有时候我们会有多个可执行文件，他们之间用到的某些功能是相同的，我们想把这些共用的
    功能做成一个库，方便大家一起共享。
  • 库中的函数可以被可执行文件调用，也可以被其他库文件调用。
    库文件又分为静态库文件和动态库文件。
  • 静态库实际上和.o文件链接没啥区别，相当于.o文件的打包。
    然而
    1. 静态库相当于直接把代码插入到生成的可执行文件中，会导致体积变大，但
      只要一个文件即可运行
    2. 只能给自己用，难以给别人用，甚至是共享着去用。
  • 动态库只在生成的可执行文件中生成“插桩“函数，当可执行文件被加载时会读取指令目录中的.dll
    文件，加载到内存中空闲的位置，并且替换相应的“插桩”指向的地址为加载后的地址，这个过程
    称为重定向。这样以后函数被调用就会跳转到动态加载的地址去。
    Sample：例如我们的ls程序

他就链接libc等库，以实现代码的共享。

CMake 中的静态库和动态库

  • CMake 除了add_executable 可以生成可执行文件外，还可以
    通过 add_library 生成库文件
    add_library 的语法和 add_library 大致相同，除了需要指定是静态库还是动态库
Sample
  1. CMakeLists.txt:
  2. cmake -B build
  1. cmake --build build

可以看到，生成了静态库libhellolib.a, 然而却并没用链接这个库

为什么呢？
这就是静态库的好处，他不需要运行时查找，因此当我们删了这个库，他还能运行

然而，当我们改为动态链接

  1. CMakeLists.txt:
  2. cmake -B build
  1. cmake --build build

可以看到，生成并链接了库libhellolib.so

当我们删了这个库时

  • 动态库有很多坑，特别是在Windows环境下。初学者创建库时，建议使用静态库。
  • 但是，他人提供的库，大多是作为动态库的，应该掌握如何使用他人的库。

创建库以后，要在某个可执行文件中使用该库，只需要：
target_link_libraries(myexec PUBLIC test) #为myexec链接库test

为什么C++需要声明

在之前的例子中，main.cpp必须声明hello()才能引用。为什么呢？

  1. 因为必须知道函数的参数和返回值类型：这样才能支持重载，隐式格式转换等特性。例如show(3), 如果声明
    void show(float), 那么编译器知道把3转换为3.0f才能调用。
  2. 让编译器知道hello这个名字是个函数，而不是变量或者类的名字；这样当写下hello()的时候，编译器才知道
    这是要调用hello这个函数，而不是创建一个叫hello的类的实例。
    顺便一提，C++是一种强烈依赖上下文信息的编程语言。

为什么需要头文件

  • 为了使用hello这个函数，我们刚才在main.cpp里声明了void hello();
    但如果另一个文件也要用hello这个函数呢？难道再声明一遍？
    加入能够只写一遍，然后自动插入到需要用hello的那些.cpp中就好了…

头文件-批量插入几行代码的硬核方式

没错，C语言的前辈也想到了，既然每个.cpp文件的这个部分一模一样，不如我把一个声明放到单独的一个文件hello.h里，然后
在需要用到hello()这个声明的地方，打上一个记号。这样在编译前把内容插进记号里，不就只用编辑一次了嘛。。。
后来，编译前替换的步骤成为了编译器的一部分，成为预处理阶段。#define定义的宏也是在这个阶段处理的。

  • 此外，在实现的文件hello.cpp中导入声明的hello.h是个好习惯，可以保证当hello.cpp被修改时，比如改成hello(int), 编译器能
    发现hello.h声明的hello()和定义的不一样，避免**“沉默的错误”**。

  • 实际上cstdio也无非是提供了printf等一系列函数声明的头文件罢了，实际的实现是在lib.so这个动态库里。
    其中这种形式表示不要在当前目录搜索，只在系统目录搜索，"hello.h"这种形式则表示优先搜索当前目录
    找不到再搜索系统目录。

头文件进阶-递归地使用头文件

  • 在C++中常常用到很多的类，和函数一样，类的声明也会被放到头文件中。
  • 有时候我们的函数声明需要使用到某些类，就需要用到声明该类的头文件，递归地#include即可。
    但这样造成一个问题，就是多个头文件都引入了MyClass.h, 那么MyClass会被重复定义两次。
两种解决方法：
  1. 法一

#ifndef CLASS_H
#define CLASS_H
// blah blah blah
#endif

这样甚至可以用<hello.h>来引用这个头文件了，因为通过target_include_directories
指定的路径会被视为与系统路径等价。

但是，加入另一个b.out也需要用hellolib这个库，难道也得指定一遍搜索路径吗？
不需要，其实我们只需要定义hellolib的头文件搜索路径，引用他的可执行文件CMake会自动
添加这个路径

由于target_link_libraries(a.out PUBLIC hellolib)中是PUBLIC, 库的属性在link的时候被自动添加。
此外如果不希望引用hellolib的可执行文件自动添加这个路径，把PUBLIC改成PRIVATE即可。

目标的一些其他选项

除了头文件搜索目录以外，还有这些选项，PUBLIC和PRIVATE对他们同理:

  1. target_include_directories(myapp PUBLIC /usr/include) # 添加头文件搜索路径
  2. target_link_libraries(myapp PUBLIC hellolib) # 添加要链接的库
  3. target_add_definitions(myapp PUBLIC MY_MACRO=1) # 添加一个宏定义
  4. target_add_definitions(myapp PUBLIC -DMY_MACRO=1) # 与MY_MACRO=1等价
  5. target_compile_options(myapp PUBLIC -fopenmp) # 添加编译器命令行选项
  6. target_soures(myapp PUBLIC other.cpp another.cpp) # 添加要编译的源文件

以及可以通过下列指令（不推荐使用），把选项加到所有接下来的目标去：

  1. include_directories(/opt/cude/include) # 添加头文件搜索路径
  2. link_directories(/opt/cuda) # 添加库文件的搜索路径
  3. add_definitions(MY_MACRO=1) # 添加一个宏定义
  4. add_compile_options(-fopenmp) # 添加编译器命令行选项

第三方库

作为纯头文件引入

C++标准库的功能不一定能满足我们的需求，难免会用到一些第三方库。
最友好的一类莫过于纯头文件库了，以下列举了一些好用的header-only库：

  1. nothings/stb - 大名鼎鼎的stb_img系列，涵盖图像，声音，字体等，只需单头文件！
  2. Neargye/magic_enum - 枚举类型的反射，如枚举转字符串等。
  3. Tencent/rapidjson - 单纯的json库，甚至没依赖STL
  4. ericniebler/range-v3 - C++20 ranges库受其启发（甚至还不如他）
  5. fmtlib/fmt - 格式化库，std::format的上位替代。

只需要把他们的include目录或头文件下载下来，然后include_directories()即可。

  • 缺点：函数直接实现在头文件中，没有提前编译，从而重复编译同样内容，编译时间长。

作为子模块引入

第二友好的方式则是作为CMake子模块引入，也就是通过 add_subdirectory
方法就是把那个项目（以fmt为例）的源代码放到工程的根目录：

Sample:
cmake_minimum_required(VERSION 3.12)    
project(hellocmake LANGUAGES CXX)

add_subdirectory(fmt)

add_executable(a.out main.cpp)
target_link_libraries(a.out PUBLIC fmt)

这些库能够很好地支持作为子模块引入：

  1. fmtlib/fmt - 格式化库，提供std::format的替代品
  2. gabime/spdlog - 能适配控制台，安卓等多后端的日志库
  3. ericniebler/range-v3 - C++20 ranges库受其启发（甚至还不如他）
  4. g-truc/glm - 模仿GLSL语法的数学矢量/矩阵库
  5. abseil/abseil-cpp - 旨在补充标准库没有的常用功能
  6. bombela/backward-cpp - 实现了C++的堆栈回溯便于调试
  7. google/googletest - 谷歌单元测试框架
  8. google/benchmark - 谷歌性能评估框架
  9. glfw/glfw - OpenGL 窗口和上下文管理
  10. libigl/libigl - 各种图形学算法大合集
